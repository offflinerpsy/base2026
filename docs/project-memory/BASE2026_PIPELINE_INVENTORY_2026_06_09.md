## 2. Public/Private Boundary

Public by default:

- attributed source records;
- source URLs;
- creator handles/profile URLs;
- publication dates when available;
- public excerpts;
- searchable passage snippets;
- public insight cards when evidence matching/review passes;
- public topic and comparison pages.

Private by default:

- raw captions;
- full transcripts;
- downloaded media/audio/video;
- local SQLite database;
- local logs;
- local staging queues;
- API keys, cookies, tokens;
- unreviewed raw source vaults.

Public export policy:

- full transcripts are not published as standalone public pages by default;
- public pages must keep attribution and source links;
- public insight cards must have `source_id` and `evidence_excerpt`;
- public source/passages JSON must not leak raw claims or full transcripts.

Relevant guard:

- `scripts/check-public-export-policy.py`

Public JSONL files are generated under:

Schema:

### Source Records

One public row per video/post:

- `source_id`
- `item_id`
- `platform`
- `source_url`
- `creator_handle`
- `creator_url`
- `published_at`
- `title`
- `excerpt`
- `topics`
- `topic_labels`
- `public_policy`

Generated by:

- `scripts/export-public-tiktok.py`

### Passages

Searchable snippets/chunks from cleaned transcript text:

- `id`
- `source_id`
- `chunk_index`
- `body`
- `topics`
- `topic_labels`
- `public_policy = search_passage`

Indexed into Meilisearch by:

- `scripts/meili-index-public.py`

### Insight Cards

Claim/viewpoint cards extracted from source evidence:

- `claim_id`
- `topic_id`
- `source_id`
- `claim_text`
- `suggested_action`
- `evidence_excerpt`
- `evidence_score`
- `public`
- `needs_review`
- `public_policy`

Important: insight cards are not generated directly from passages during public page generation. They come from the local knowledge database tables `claims` and `claim_evidence`.

## 4. Why Some Public Insight Cards Are Empty

Sample checked source:

- page: `/knowledge/sources/tiktok-video-7647909694559767840.html`
- source id: `tiktok:build_in_public:7647909694559767840`
- related passages: 1
- insight cards: 0
- public insight cards: 0

Dataset-wide check:

- total sources: 957
- sources without any insight cards: 173
- sources without public insight cards: 305
- sources without insight cards but with passages: 170

Root cause:

1. `scripts/generate-public-pages.py` only displays insight cards already present in `public-data/tiktok/insight_cards.jsonl`.
2. `insight_cards.jsonl` is created by `scripts/export-public-tiktok.py`.
3. `export-public-tiktok.py` creates cards only from this SQLite query:
   - `claim_evidence`
   - joined to `claims`
   - filtered by video id.
4. If a video has transcript/passages but no extracted claim rows, its source page has no insight cards.
5. If claim rows exist but fail review/evidence thresholds, cards may exist privately but not be public.

Conclusion:

- This is not a source-page rendering bug.
- It is a pipeline gap: claim/insight extraction was not completed for every transcript.
- Empty cards are acceptable only when the source truly has no useful claim. For SEO/GEO/AEO creator videos, most empty pages probably need a claim extraction pass.

Current schedule:

- daily 03:30 Minsk time
- daily 15:30 Minsk time

Current command:

- `scripts/hermes-tiktok-refresh.ps1 -CheckOnly -PlaylistEnd 50`

Current behavior:

- inventory/check only;
- detects or refreshes candidate state;
- does not complete captions/ASR;
- does not polish transcripts;
- does not extract claims;
- does not import into KB;
- does not package;
- does not deploy.

Conclusion:

- Current automation is useful as a safety check.
- It is not a finished update pipeline.
- The project still needs a reviewed local worker pipeline that can process and deploy new TikToks after QA gates pass.

## 8. Recommended Target Pipeline

The target should be a local-first worker controlled by Codex/Hermes, not a paid SaaS dependency.

### Stage A: Fast Inventory

No LLM.

Tasks:

- check configured creators;
- pull latest post IDs/URLs;
- dedupe against local KB;
- report new candidates.

Model:

- none.

### Stage B: Caption / Metadata Extraction

No LLM unless extraction fails.

Tasks:

- collect platform caption/title/date/source URL;
- isolate the active video page;
- reject mixed recommended-video content.

Model:

- none.

### Stage C: ASR Fallback

Use only when captions are missing/incomplete.

Recommended local ASR:

- `faster-whisper`

Model:

- `medium.en` or `large-v3-turbo` depending local GPU/quality tradeoff.

### Stage D: Faithful Transcript Cleanup

Deterministic first, local LLM only when needed.

Allowed:

- punctuation;
- paragraph breaks;
- obvious casing fixes;
- remove UI noise;
- preserve speaker wording/order.

Forbidden:

- summaries instead of transcripts;
- new facts;
- Russian translation;
- marketing rewrite;
- invented connective tissue.

Candidate local LLM:

- Gemma 3/4 class model if locally available and quality passes;
- Qwen-class instruct model is also worth testing for structured extraction;
- GPT-5.4 only for small failed batches or QA escalation.

### Stage E: Claim / Insight Extraction

This is the missing layer for empty cards.

For each cleaned transcript:

- extract only useful SEO/GEO/AEO/AI visibility claims;
- assign topic;
- produce concise `claim_text`;
- produce `suggested_action` only if directly supported;
- quote or reference evidence excerpt;
- keep all new cards `pending` unless evidence match passes.

Preferred route:

1. Create claim batches from all transcripts that have passages but no claims.
2. Use a small/local model for first pass.
3. Run deterministic evidence matching.
4. Auto-promote only high-confidence cards.
5. Keep weak/uncertain cards as private `needs_review`.

### Stage F: Public Export

No LLM.

Tasks:

- rebuild SQLite;
- export public JSONL;
- validate no transcript/raw leakage;
- generate static pages;
- generate sitemap.

### Stage G: Deploy Gate

No LLM unless summarizing QA.

Tasks:

- package release;
- reindex Meilisearch;
- deploy to VPS;
- run live smoke tests;
- update repo memory.

## 9. Proposed Worker Split

Worker 1: inventory/dedupe

- no LLM
- fast, scheduled
- safe to run twice daily

Worker 2: transcript worker

- caption first
- `faster-whisper` fallback
- deterministic cleanup
- local LLM only for damaged text

Worker 3: insight-card worker

- takes polished transcripts and existing passages;
- creates pending claim cards;
- evidence-matches claims;
- promotes only safe public cards.

Worker 4: exporter/deployer

- no LLM;
- validates public policy;
- deploys only if data changed and QA passes.

## 11. SEO / GEO / AEO Usefulness Layer

The SEO skills should be used after evidence extraction, not before source fidelity is established.

Use SEO/GEO/AEO logic for:

- topic normalization;
- intent/category labels;
- claim usefulness scoring;
- actionability scoring;
- comparison pages;
- internal linking;
- title/meta generation for indexable aggregate pages;
- schema fields.

Do not use SEO skills to:

- rewrite transcripts;
- invent stronger claims;
- inflate weak evidence;
- turn raw captions into SEO content without source support.

## 13. Bottom Line

Base2026 currently has a working public search/product shell, safe excerpt-first source pages, and repeatable deploy mechanics.

The main unfinished engineering layers are:

- full automated intake/update pipeline;
- local ASR and transcript cleanup hardening;
- backfill claim/insight extraction for sources with passages but no cards;
- reviewed public promotion workflow;
- GitHub/publication staging.
